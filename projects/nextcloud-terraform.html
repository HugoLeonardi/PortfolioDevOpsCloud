📌 Présentation du projet

Ce projet déploie automatiquement Nextcloud — une solution de stockage et partage de fichiers open-source — sur AWS, en suivant une approche Infrastructure as Code (IaC) avec Terraform.

L'objectif est double : démontrer la maîtrise du provisionnement cloud automatisé, et appliquer les bonnes pratiques de sécurité et d'optimisation des coûts dès la conception de l'architecture.

  • Infrastructure entièrement reproductible, versionnée dans Git
  • Déploiement ou destruction en une commande (apply / destroy)
  • Aucune configuration manuelle post-déploiement grâce au script userdata.sh
  • Architecture 3-tiers respectant les principes de séparation des responsabilités
ℹ️

Contexte : projet réalisé dans le cadre du BTS SIO SISR (épreuve E4/E5), région AWS eu-west-3 (Paris). Stack : Terraform ≥ 1.0 · AWS Provider ~5.0.

🏗️ Architecture 3-Tiers

L'infrastructure est organisée en trois couches indépendantes, isolées par des Security Groups et des sous-réseaux dédiés. Seul le serveur web est exposé à Internet ; la base de données et le stockage restent inaccessibles depuis l'extérieur.

🔒 Sécurité

La sécurité est intégrée à l'architecture, pas ajoutée après coup. Six mesures concrètes ont été implémentées, couvrant l'isolation réseau, la gestion des identités et le chiffrement des données.

🌐
Isolation réseau (VPC)

Infrastructure déployée dans un VPC dédié (10.0.0.0/16). RDS placé dans des subnets privés sans route vers Internet. Seul l'EC2 est dans le subnet public.

🛡️
Security Groups — Moindre privilège

Le Security Group de RDS n'autorise que le port 3306 en provenance du Security Group EC2 (pas d'une IP). Toute autre connexion est refusée par défaut.

🔑
IAM Role — Sans credentials stockés

L'EC2 accède à S3 via un IAM Role (Instance Profile), sans clé d'accès hardcodée. Les permissions sont limitées aux actions nécessaires (GetObject, PutObject, DeleteObject, ListBucket) sur le bucket spécifique uniquement.

🔐
Chiffrement S3 (AES-256)

Chiffrement serveur activé sur le bucket (SSE-S3 AES-256). Accès public entièrement bloqué via aws_s3_bucket_public_access_block (4 paramètres à true).

💽
Volume EC2 chiffré (gp3)

Le disque racine de l'instance EC2 est chiffré (encrypted = true), en type gp3 pour de meilleures performances sans surcoût.

⚠️
SSH — À restreindre en production

En environnement de lab, le port 22 est ouvert sur 0.0.0.0/0. En production, il faut remplacer par VOTRE_IP/32 ou utiliser AWS Systems Manager Session Manager (sans port SSH ouvert).

Extraits de code clés

① Security Group RDS — source = SG de l'EC2, pas une IP

terraform 
# MySQL uniquement depuis le serveur EC2
ingress {
  from_port       = 3306
  to_port         = 3306
  protocol        = "tcp"
  security_groups = [aws_security_group.ec2.id]
  # Référence le SG, pas une IP → plus robuste
}

② IAM Policy — droits limités au bucket spécifique

terraform 
Resource = [
  aws_s3_bucket.nextcloud_storage.arn,
  "${aws_s3_bucket.nextcloud_storage.arn}/*"
  # Accès limité à CE bucket uniquement
]

③ S3 — blocage complet de l'accès public

terraform 
block_public_acls       = true
block_public_policy     = true
ignore_public_acls      = true
restrict_public_buckets = true

⚙️ Points d'implémentation notables

Userdata — Installation automatique au premier démarrage

Le script userdata.sh est passé à l'EC2 via templatefile(), ce qui permet d'injecter dynamiquement les variables Terraform (endpoint RDS, nom du bucket, région…) dans le script shell. Nextcloud est installé et configuré sans aucune intervention manuelle.

terraform 
user_data = templatefile("userdata.sh", {
  db_host     = aws_db_instance.nextcloud.address
  s3_bucket   = aws_s3_bucket.nextcloud_storage.bucket
  aws_region  = var.aws_region
  # ... autres variables injectées automatiquement
})

Dépendance explicite EC2 → RDS

L'EC2 démarre uniquement quand RDS est prêt (provisionnement ~10 min), évitant les erreurs de connexion à la base de données au démarrage.

terraform 
depends_on = [aws_db_instance.nextcloud]
# Terraform attend que RDS soit en état "available"
# avant de lancer l'EC2

Unicité du nom S3 avec random_id

Les noms de buckets S3 sont globalement uniques sur AWS. Un suffixe aléatoire est généré automatiquement pour éviter les conflits lors de redéploiements.

terraform 
resource "random_id" "bucket_suffix" {
  byte_length = 4
}

bucket = "${var.project_name}-storage-${random_id.bucket_suffix.hex}"
# → ex: nextcloud-sio-storage-a1b2c3d4

🚀 Déploiement

L'infrastructure complète se déploie en 4 étapes, en moins de 15 minutes.

1
Prérequis
Installer Terraform ≥ 1.0 et l'AWS CLI, configurer les credentials (aws configure) et créer une paire de clés SSH dans la console AWS.
2
Initialisation
terraform init — télécharge le provider AWS et initialise le backend local.
3
Plan & Apply
terraform plan pour vérifier les ressources à créer, puis terraform apply et confirmer avec yes. ⏱ ~10-15 min (RDS est long à provisionner).
4
Récupération des outputs
terraform output affiche l'URL Nextcloud, la commande SSH, l'endpoint RDS et le nom du bucket S3.
